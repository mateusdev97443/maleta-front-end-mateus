# Maleta Front-end Mateus

Repositório profissional de consulta, prática e evolução para o estudante de Front-end Mateus.

A Maleta centraliza manuais, snippets, checklists, prompts, projetos guiados e registros de evolução para consolidar fundamentos de Front-end com organização, acessibilidade, clareza e prática real.

## Objetivo

Manter uma base de estudo e execução para evoluir em Front-end de forma progressiva, começando por HTML, CSS e JavaScript vanilla antes de avançar para ferramentas mais complexas.

A Maleta deve ajudar a:

- consultar conceitos essenciais antes de implementar;
- praticar fundamentos em projetos pequenos e revisáveis;
- reaproveitar padrões somente depois de entendê-los;
- revisar entregas com critérios claros de qualidade;
- registrar decisões, aprendizados e próximos passos;
- evitar acúmulo de arquivos soltos, duplicados ou sem contexto.

## Como usar esta maleta

1. Consulte o `01-manual-de-consulta/` antes de implementar uma tecnologia ou revisar conceitos.
2. Use os exercícios, mini-desafios e perguntas de revisão para testar entendimento.
3. Reaproveite padrões de `02-snippets-e-componentes/` somente depois de entender o que eles fazem.
4. Crie projetos a partir de `03-fabrica-de-projetos/`, começando pelo template base vanilla quando fizer sentido.
5. Valide entregas com `04-checklists-de-qualidade/` antes de considerar um projeto finalizado.
6. Use `05-prompts-codex/` para pedir revisão, explicação, refatoração e planejamento com mais precisão.
7. Registre aprendizados e evolução em `06-dossie-de-evolucao/`.
8. Mova materiais obsoletos, substituídos ou pausados para `99-arquivo/`, mantendo histórico sem poluir a base principal.

## Princípios da maleta

- **Clareza antes de volume:** cada arquivo deve resolver uma necessidade real.
- **Base vanilla primeiro:** HTML, CSS e JavaScript sem frameworks até consolidar fundamentos.
- **Mobile-first:** todo projeto nasce funcionando bem em telas pequenas.
- **Acessibilidade como padrão:** semântica, contraste, foco visível e navegação por teclado.
- **Evolução registrada:** aprendizados e decisões importantes devem ficar documentados.
- **Escopo explícito:** cada fase, projeto ou área deve deixar claro o que faz e o que não faz.
- **Manutenção contínua:** áreas de suporte podem evoluir sem invalidar fases já concluídas.

## Fase concluída, área em evolução e backlog

- **Fase concluída:** entrega fechada, revisada e aceita dentro do escopo definido para aquela etapa.
- **Área em evolução:** espaço de suporte que pode crescer sem alterar o status das fases já concluídas.
- **Backlog:** ideia, categoria ou melhoria planejada que ainda não deve ser tratada como conteúdo pronto.

Uma área estar em evolução não significa erro. Significa que ela existe para receber melhorias controladas conforme a Maleta amadurece.

## Critérios de qualidade da Maleta

Antes de considerar um material ou projeto como pronto, verifique se ele atende aos critérios abaixo:

- objetivo claro e alinhado com a evolução Front-end;
- escopo explícito, sem prometer o que não foi implementado;
- HTML semântico quando houver página;
- `lang="pt-BR"`, `viewport` e título adequado em páginas HTML;
- CSS mobile-first, legível e sem dependências externas;
- foco visível para elementos interativos;
- contraste e espaçamento suficientes para leitura confortável;
- JavaScript vanilla organizado em funções claras quando houver interação;
- tratamento de loading, erro, vazio e sucesso quando houver dados dinâmicos;
- uso seguro do DOM, evitando `innerHTML` com dados externos;
- ausência de tokens, chaves privadas e dependências desnecessárias;
- documentação de como usar, testar e revisar.

## Como revisar antes de avançar

1. Leia o README principal e confirme se o status da área está claro.
2. Abra a página ou projeto localmente quando houver HTML executável.
3. Confira links relativos e caminhos principais.
4. Revise acessibilidade básica: headings, labels, foco visível, contraste e teclado.
5. Revise escopo: sem frameworks, sem build, sem dependências externas e sem promessas não entregues.
6. Use os checklists de `04-checklists-de-qualidade/` para registrar a revisão.
7. Registre decisões relevantes no dossiê quando a mudança alterar direção, padrão ou aprendizado.
